<task type="auto">
  <name>Task 1: Add sessions and secrets tables to database schema</name>
  <files>lib/db/schema.ts, lib/db/index.ts</files>
  <action>
Add both new tables to `lib/db/schema.ts` after the existing `settings` table.

Sessions table (for auth):
```typescript
export const sessions = sqliteTable("sessions", {
  id: text("id").primaryKey().$defaultFn(() => crypto.randomUUID()),
  token: text("token").notNull().unique(),
  expiresAt: text("expires_at").notNull(),
  createdAt: text("created_at").notNull().$defaultFn(now),
});
```

Secrets table (for encrypted API keys):
```typescript
export const secrets = sqliteTable("secrets", {
  id: text("id").primaryKey().$defaultFn(() => crypto.randomUUID()),
  name: text("name").notNull().unique(),
  encryptedValue: text("encrypted_value").notNull(),
  iv: text("iv").notNull(),
  authTag: text("auth_tag").notNull(),
  description: text("description"),
  createdAt: text("created_at").notNull().$defaultFn(now),
  updatedAt: text("updated_at").notNull().$defaultFn(now),
});
```

Add inferred types at the bottom of the file for both tables:
```typescript
export type Session = typeof sessions.$inferSelect;
export type NewSession = typeof sessions.$inferInsert;

export type Secret = typeof secrets.$inferSelect;
export type NewSecret = typeof secrets.$inferInsert;
```

In `lib/db/index.ts`, add `Session`, `NewSession`, `Secret`, and `NewSecret` to the re-exported types.

NOTE: We do NOT need a `users` table. This is single-user auth. The password hash is stored in the existing `settings` table under the key `"passwordHash"`. This avoids schema complexity for a single-user system and fits the existing settings pattern.

<task type="auto">
  <name>Task 2: Build password hashing and session management libraries</name>
  <files>lib/auth/password.ts, lib/auth/session.ts</files>
  <action>
Install bcryptjs (pure JS, no native deps — avoids build issues in Next.js): `npm install bcryptjs && npm install -D @types/bcryptjs`

Create `lib/auth/password.ts`:
- `hashPassword(plain: string): string` — uses `bcryptjs.hashSync(plain, 12)` (sync because better-sqlite3 is sync)
- `verifyPassword(plain: string, hash: string): boolean` — uses `bcryptjs.compareSync(plain, hash)`
- `isPasswordSet(): boolean` — checks if settings table has a `"passwordHash"` key (import db and settings from `@/lib/db` and use `db.select().from(settings).where(eq(settings.key, "passwordHash")).get()`)

Create `lib/auth/session.ts`:
- `import { randomBytes } from "crypto"`
- `SESSION_DURATION = 24 * 60 * 60 * 1000` (24 hours default)
- `REMEMBER_ME_DURATION = 30 * 24 * 60 * 60 * 1000` (30 days)
- `SESSION_COOKIE_NAME = "homelabarr_session"`
- `createSession(rememberMe: boolean = false): string` — generates token via `randomBytes(32).toString("hex")`, inserts into sessions table with appropriate expiresAt, returns the token
- `validateSession(token: string): Session | null` — looks up session by token, checks if expiresAt > now. If expired, delete it and return null. If valid, return the session row.
- `deleteSession(token: string): void` — deletes session row by token
- `cleanExpiredSessions(): void` — deletes all sessions where expiresAt < now (use `lt` from drizzle-orm)
- `getSessionFromCookies(): Session | null` — reads the cookie from `cookies()` (next/headers), extracts token, calls validateSession. Import `cookies` from `"next/headers"`.
- `setSessionCookie(token: string, rememberMe: boolean): void` — sets httpOnly cookie with proper attributes. Use `cookies().set()` with: `{ httpOnly: true, secure: process.env.NODE_ENV === "production", sameSite: "lax", path: "/", maxAge: rememberMe ? REMEMBER_ME_DURATION / 1000 : SESSION_DURATION / 1000 }`
- `clearSessionCookie(): void` — deletes the session cookie

IMPORTANT: In Next.js 16, `cookies()` is async — you must `await cookies()`. All functions that call `cookies()` must be async.

IMPORTANT: When comparing dates for expiry, compare ISO strings directly since ISO 8601 strings are lexicographically sortable: `session.expiresAt > new Date().toISOString()` works correctly.
  </action>

  <done>password.ts exports hashPassword, verifyPassword, isPasswordSet. session.ts exports createSession, validateSession, deleteSession, getSessionFromCookies, setSessionCookie, clearSessionCookie. Both files compile without TypeScript errors.</done>

Create `app/api/auth/session/route.ts` with a GET handler:

- GET handler:
  - Call `isPasswordSet()` — if false, return `{ authenticated: false, setupRequired: true }` (no password has been set yet, user needs initial setup)

- Always return 200 status (this is a status check, not a protected resource)

This endpoint is used by the login page (Plan 03) to determine whether to show login form or initial setup form.
