<task type="auto">
  <name>Task 1: Build AES-256-GCM encryption module</name>
  <files>lib/crypto/secrets.ts</files>
  <action>
Create `lib/crypto/secrets.ts` using Node.js built-in `crypto` module:

- `import { createCipheriv, createDecipheriv, randomBytes, scryptSync } from "crypto"`
- `import { readFileSync, writeFileSync, existsSync, mkdirSync } from "fs"`
- `import { dirname } from "path"`

The encryption key source: check `process.env.ENCRYPTION_SECRET` first. If not set, auto-generate a 32-byte hex key and persist it to `./data/.encryption-key` (same data dir as the DB). This file-based fallback ensures the key survives restarts without requiring env vars.

Functions:
- `getEncryptionKey(): Buffer` — derives a 32-byte key using `scryptSync(secret, "homelabarr-secrets-salt", 32)` where `secret` is from env or the file. Cache the result in a module-level variable so scrypt only runs once.
- `encrypt(plaintext: string): { encrypted: string, iv: string, authTag: string }` — uses `aes-256-gcm` with a random 16-byte IV. Returns hex-encoded strings for all three values.
- `decrypt(encrypted: string, iv: string, authTag: string): string` — reverses the encryption. Returns the plaintext string.

CRITICAL: The `getEncryptionKey` function must handle the auto-generation path carefully:
1. Check `process.env.ENCRYPTION_SECRET`
2. If not set, check if `./data/.encryption-key` exists
3. If file doesn't exist, generate `randomBytes(32).toString("hex")`, write to file, use it
4. If file exists, read it
5. Pass the secret through scryptSync for proper key derivation

Add `data/.encryption-key` to `.gitignore` if not already ignored (check first — the data directory may already be gitignored).

<task type="auto">
  <name>Task 2: Build secrets CRUD API routes</name>
  <files>app/api/secrets/route.ts, app/api/secrets/[id]/route.ts</files>
  <action>
Create `app/api/secrets/route.ts`:

GET handler — list all secrets:
- Import `db` from `@/lib/db` and `secrets` from `@/lib/db/schema`
- Query all rows from secrets table
- Return them with values MASKED: replace encryptedValue/iv/authTag with a `masked: true` flag and show only `id`, `name`, `description`, `createdAt`, `updatedAt`
- Never return encrypted values in list responses

POST handler — create a secret:
- Accept `{ name: string, value: string, description?: string }` in request body
- Validate name and value are non-empty strings
- Import `encrypt` from `@/lib/crypto/secrets`
- Call `encrypt(value)` to get encrypted, iv, authTag
- Insert into secrets table
- Return the created secret (masked — no encrypted value in response)

Create `app/api/secrets/[id]/route.ts`:

Use the same RouteContext pattern as existing `app/api/settings/[key]/route.ts`:
```typescript
type RouteContext = { params: Promise<{ id: string }> };
```

GET handler — get a single secret's decrypted value:
- Look up by id
- Import `decrypt` from `@/lib/crypto/secrets`
- Call `decrypt(row.encryptedValue, row.iv, row.authTag)`
- Return `{ id, name, value: decryptedValue, description }` — this is the ONLY endpoint that returns the actual value, used when editing

PUT handler — update a secret:
- Accept `{ name?: string, value?: string, description?: string }`
- If value is provided, re-encrypt it with `encrypt(value)`
- Build an update object with the provided fields plus new updatedAt timestamp
- Update the row
- Return the updated secret (masked)

DELETE handler — delete a secret:
- Delete by id using `eq(secrets.id, id)`
- Return `{ success: true }`

NOTE: These API routes are NOT yet protected by auth middleware (that comes in Plan 03). For now they are open endpoints. Plan 03's middleware will protect all /api/* routes except /api/auth/*.

IMPORTANT: All handlers must be wrapped in try/catch returning proper error responses with NextResponse.json, following the pattern in existing API routes.
