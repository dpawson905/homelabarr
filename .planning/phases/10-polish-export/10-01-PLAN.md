<objective>
Build the server-side export/import engine and API endpoints for dashboard configuration portability.

Purpose: Enables users to back up, restore, and migrate their entire dashboard configuration as a single JSON file. This is the data layer that the UI (Plan 02) will consume.

Output: Two API routes (GET /api/config/export, POST /api/config/import) backed by export/import logic modules with a versioned JSON schema.
</objective>

<context>
@.planning/PROJECT.md
@.planning/ROADMAP.md
@.planning/STATE.md

The database has these tables (all in lib/db/schema.ts):
- boards: id, name, icon, position, createdAt, updatedAt
- widgets: id, boardId, type, x, y, w, h, createdAt, updatedAt
- widgetConfigs: id, widgetId, config (JSON string), createdAt, updatedAt
- apps: id, name, url, icon, description, statusCheckEnabled, statusCheckInterval, createdAt, updatedAt
- settings: key, value, updatedAt
- sessions: id, token, expiresAt, createdAt (EXCLUDED from export)
- secrets: id, name, encryptedValue, iv, authTag, description, createdAt, updatedAt (EXCLUDED from export)

DB access is synchronous via better-sqlite3. Use `.all()` for lists, `.get()` for single rows.
Import uses Drizzle's `db.transaction()` for atomicity.
Existing widget types: app-links, search, clock, notes, weather, calendar, rss-feed, system-stats, docker, media-server, media-management, download-client, media-requests, dns, proxmox, home-assistant, uptime-kuma.
</context>

3. Queries all apps
4. Queries all settings, filtering out security-sensitive keys

  <action>
Create `lib/export/import.ts` with an `importConfig(data: HomelabarrExport, mode: "merge" | "replace")` function:

**Validation first:**
- Check `data.version === 1` (reject unknown versions with a clear error)
- Validate that `data.boards` is an array, `data.apps` is an array, `data.settings` is an object
- For each board's widgets, validate that `type` is a non-empty string (don't reject unknown widget types -- just import them, the renderer already falls back to WidgetPlaceholder)

**Replace mode** (wrapped in a single `db.transaction()`):
1. Delete all existing widgetConfigs, widgets, boards, apps
2. Delete all settings EXCEPT `passwordHash` (preserve auth)
3. Insert boards from the export, generating new UUIDs via `crypto.randomUUID()`
4. For each board, insert its widgets with new UUIDs, referencing the new board ID
5. For each widget, insert its config as a JSON string
6. Insert apps with new UUIDs
7. Insert settings (skip `passwordHash` and `defaultBoardId`)
8. Set `defaultBoardId` to the first imported board's new ID

**Merge mode** (also transactional):
1. Insert boards (always create new -- no name-based dedup, user might want duplicates)
2. Insert all widgets under their respective new boards
3. For apps: if an app with the same name AND url already exists, skip it; otherwise insert
4. For settings: only insert keys that don't already exist (don't overwrite)
5. Keep existing `defaultBoardId` unchanged

Return an `ImportResult` object: `{ boardsImported: number, widgetsImported: number, appsImported: number, settingsImported: number, warnings: string[] }`. Add warnings for skipped apps (merge mode) or any validation issues.

**API Routes:**

`app/api/config/export/route.ts` -- GET handler:
- Call `buildExport()`
- Return the JSON with `Content-Disposition: attachment; filename="homelabarr-export-YYYY-MM-DD.json"` header
- Set `Content-Type: application/json`

`app/api/config/import/route.ts` -- POST handler:
- Accept JSON body with `{ config: HomelabarrExport, mode: "merge" | "replace" }`
- Validate mode is one of the two allowed values
- Call `importConfig(config, mode)`
- Return the `ImportResult` as JSON
- Catch errors and return 400 for validation errors, 500 for unexpected failures
