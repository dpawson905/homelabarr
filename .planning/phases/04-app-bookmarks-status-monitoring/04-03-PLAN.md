<task type="auto">
  <name>Task 1: Integrate health status polling into AppLinksWidget and AppCard</name>
  <files>
    components/widgets/app-links-widget.tsx
    components/widgets/app-card.tsx
  </files>
  <action>
Update AppLinksWidget (created in Plan 01) to:
- After loading apps, immediately fetch GET /api/apps/health to get all statuses
- Store statuses in state as a Record&lt;string, HealthCheckResult&gt; mapping app ID to health result
- Set up a polling interval (default 60 seconds) using useEffect + setInterval that re-fetches /api/apps/health
- Clean up the interval on unmount
- Pass the relevant health status to each AppCard as a prop
- The health fetch shape from Plan 02 is: `{ statuses: { [appId]: { status, latency, statusCode, checkedAt } } }`

Update AppCard (created in Plan 01) to:
- Accept an optional `health` prop with shape `{ status: "online" | "offline" | "degraded" | "disabled", latency?: number, statusCode?: number | null, checkedAt?: string }`
- Replace the gray placeholder dot with a real status indicator:
  - "online" = green dot (bg-emerald-500) with a subtle pulse animation
  - "offline" = red dot (bg-red-500)
  - "degraded" = yellow/amber dot (bg-amber-500)
  - "disabled" or undefined = gray dot (bg-muted-foreground/30)
- Show latency as a small text on hover (e.g., "142ms") via a title attribute on the status dot
- The status dot should be positioned in the top-left or bottom-right corner of the card, small (size-2 or size-2.5) with a ring to make it pop

Keep the polling lightweight -- the server-side cache (from Plan 02) means rapid polls don't hammer the actual services.
  </action>
  <verify>
Run `npx tsc --noEmit` -- no type errors. Run `npm run build` -- builds successfully.

1. Add an app with statusCheckEnabled=1 and a valid URL (e.g., https://google.com) -- green dot appears within a few seconds
2. Add an app with statusCheckEnabled=1 and an invalid URL -- red dot appears
3. Add an app with statusCheckEnabled=0 -- gray dot
4. Wait 60+ seconds -- statuses refresh (check browser network tab for /api/apps/health polls)
  </verify>
  <done>
App cards show live green/red/yellow/gray status indicators based on real HTTP health checks. Statuses auto-refresh on a 60-second polling interval. Latency is visible on hover.
  </done>

<task type="auto">
  <name>Task 2: Build SearchWidget and wire it into the widget system</name>
  <files>
    components/widgets/search-widget.tsx
    components/widget-renderer.tsx
    components/add-widget-dialog.tsx
  </files>
  <action>
Create SearchWidget as a client component that:
- Fetches all apps from GET /api/apps on mount (same pattern as AppLinksWidget)
- Renders an inline search input using the existing Command (cmdk) components from components/ui/command.tsx
- This is NOT a dialog/modal -- it's an inline embedded command palette that lives inside the widget card
- Uses Command, CommandInput, CommandList, CommandEmpty, CommandGroup, CommandItem from the existing ui/command.tsx
- CommandInput is always visible at the top of the widget
- CommandList shows below the input, displaying matching apps
- When the search input is empty, show all apps (up to first 10, with a "and N more..." note if there are more)
- Filtering is client-side on name, description, and URL fields (cmdk handles this automatically via its built-in filtering)
- Each CommandItem shows: the app icon (or first-letter avatar matching AppCard's style), the app name, and a truncated URL in muted text
- Selecting an item (click or Enter) opens the app URL in a new tab via `window.open(url, '_blank', 'noopener,noreferrer')`
- Style the widget to fit within the grid card: full width/height, rounded-lg border border-border bg-card, with the input at top and scrollable results below
- Empty state when no apps exist: "No apps found. Add apps via the App Links widget."

Update widget-renderer.tsx to add a case for type "search" that renders SearchWidget.

Update add-widget-dialog.tsx:
- Add a "Search" entry to the WIDGET_TYPES array: `{ type: "search", name: "Search", description: "Search across all your apps" }`. Place it second in the list (after app-links) since it's a primary utility widget.
- Update the handleAddWidget function to support type-specific default sizes. For "search" type use `{ w: 4, h: 2 }`, keep `{ w: 3, h: 2 }` as the default for all other types. Implement this as a simple size lookup object keyed by widget type.
  </action>
