  <action>
Create a WidgetRenderer component that receives widget type, widget id, and config as props. It uses a switch/map to dispatch to the correct widget component based on type. For "app-links", render AppLinksWidget. For all other unimplemented types, fall back to the existing WidgetPlaceholder.

Update widget-grid.tsx to import and use WidgetRenderer instead of WidgetPlaceholder. Pass the widget's type, id, and config object through.

Build AppLinksWidget as a client component that:
- Fetches all apps from GET /api/apps on mount (the existing API returns `{ apps: [...] }`)
- Renders a grid of AppCard components inside a scrollable container
- Shows a header bar with "Apps" title and an "Add App" button (+ icon)
- If no apps exist, shows a friendly empty state with an "Add your first app" prompt
- Clicking "Add App" opens the AppFormDialog (built in Task 2)
- Includes a useCallback for refreshing the app list after add/edit/delete operations
- Uses polling or manual refresh (not websockets) -- apps refresh on dialog close

Build AppCard as a presentational component that:
- Renders a clickable card that opens the app URL in a new tab (target="_blank", rel="noopener noreferrer")
- Shows the app icon (if set, render as a small image/emoji; if not, show the first letter of the app name as a colored avatar)
- Shows the app name (truncated if long) and description (1-line truncated)
- Has a small status indicator dot in the corner (just render it as a gray dot for now -- Plan 02 adds the real health check coloring)
- On hover, shows a subtle edit button (pencil icon) and delete button (trash icon) in the top-right corner
- Edit button opens AppFormDialog in edit mode (passed via callback prop)
- Delete button calls DELETE /api/apps/[id] after confirmation, then triggers refresh

Style all components using the existing oklch color variables and Tailwind classes. Match the aesthetic of existing components (card backgrounds, border colors, text sizing from board-dialogs.tsx and widget-placeholder.tsx). Use HugeiconsIcon for all icons (PlusSignIcon, PencilEdit01Icon, Delete02Icon, Link04Icon or similar from @hugeicons/core-free-icons).

<task type="auto">
  <name>Task 2: Build Add/Edit App form dialog</name>
  <files>
    components/app-form-dialog.tsx
  </files>
  <action>
Create AppFormDialog as a client component using the existing Dialog, DialogContent, DialogHeader, DialogTitle, DialogDescription, DialogFooter pattern from board-dialogs.tsx.

The dialog accepts props:
- open: boolean
- onOpenChange: (open: boolean) => void
- onSaved: () => void (callback to refresh parent)
- app?: App | null (if provided, edit mode; if null/undefined, create mode)

Form fields:
- Name (required, text Input)
- URL (required, text Input with placeholder "https://...")
- Icon (optional, text Input -- for now just a text field where user can type an emoji or icon name; icon library comes in Phase 10)
- Description (optional, Textarea or Input)
- Enable Status Check (Switch/Checkbox toggle)
- Check Interval (number Input, only shown when status check is enabled, default 300 seconds, label "Check every N seconds")

On submit:
- Create mode: POST to /api/apps with the form data
- Edit mode: PATCH to /api/apps/[id] with changed fields
- Show loading state on submit button ("Creating..."/"Saving...")
- On success: call onSaved(), close dialog
- On error: show inline error message (same pattern as board-dialogs.tsx)

Use the existing UI components: Dialog, Button, Input, Label. Use the Switch component from ui/switch.tsx for the status check toggle. Follow the exact same form patterns established in board-dialogs.tsx (useState for each field, useEffect to reset on open, error state, loading state).

URL field should have basic validation: must start with http:// or https://.
  </action>
  <verify>
Run `npx tsc --noEmit` -- no type errors. Open the app-links widget, click "Add App", fill in name + URL, submit. App appears in the widget. Click edit on the app card, change the name, save. Name updates. Verify via `curl http://localhost:3000/api/apps` that apps are persisted.
  </verify>
  <done>
AppFormDialog creates and edits apps with all fields. Validation prevents submission without name/URL. Created/edited apps appear in the app-links widget after dialog closes.
