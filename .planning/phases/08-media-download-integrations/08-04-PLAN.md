Infrastructure from Plan 01:
- `fetchService(options)` handles authenticated HTTP requests
- `getServiceConnection(config)` resolves serviceUrl + apiKey
- SABnzbd uses "query-apikey" auth type (API key as query param)
- qBittorrent requires special handling (cookie-based auth, not header-based)

qBittorrent Web API v2 reference:
- Auth: POST /api/v2/auth/login with body "username=admin&password=adminadmin" (form-encoded)
  - Returns Set-Cookie header with SID cookie on success
  - Body text is "Ok." on success, "Fails." on failure
- Torrent list: GET /api/v2/torrents/info?filter=all (or filter=downloading, filter=completed, etc.)
  - Returns array of torrent objects:
    - name, hash, size (bytes), progress (0.0-1.0), dlspeed (bytes/sec), upspeed (bytes/sec)
    - state: "downloading", "uploading", "stalledDL", "stalledUP", "pausedDL", "pausedUP", "queuedDL", "queuedUP", "checkingDL", "error", "missingFiles", "moving", "forcedDL"
    - eta: seconds remaining (-1 if unknown)
    - added_on: unix timestamp
- NOTE: qBittorrent auth needs username/password, NOT an API key. The widget config should store username in config and password in secrets.

SABnzbd API reference:
- Auth: API key as query param: ?apikey=xxx&output=json
- Queue: GET /api?mode=queue&output=json&apikey=xxx
  - Returns { queue: { slots: [...], speed: "2.5 M", timeleft: "0:15:30", ... } }
  - Each slot: { filename, mb (total MB), mbleft (MB remaining), percentage (0-100), status (Downloading/Paused/Queued), timeleft }
- History (completed): GET /api?mode=history&output=json&apikey=xxx&limit=20
  - Returns { history: { slots: [...] } }
  - Each slot: { name, bytes, status (Completed/Failed), completed (unix timestamp) }

**Types file: `app/api/widgets/download-client/types.ts`**

```typescript
export interface DownloadItem {
  id: string
  name: string
  status: "downloading" | "paused" | "queued" | "completed" | "failed" | "stalled" | "seeding" | "checking"
  progressPercent: number   // 0-100
  sizeTotal: number         // bytes
  sizeRemaining: number     // bytes
  downloadSpeed: number     // bytes/sec (0 if not actively downloading)
  uploadSpeed?: number      // bytes/sec (torrents only)
  eta: string               // formatted time remaining, empty if unknown
  addedAt?: string          // ISO date string
}

export interface DownloadSummary {
  totalSpeed: string        // formatted overall download speed
  totalItems: number
  activeItems: number
}

export interface DownloadClientResponse {
  items: DownloadItem[]
  summary: DownloadSummary
  serviceType: "qbittorrent" | "sabnzbd"
}
```

**API route: `app/api/widgets/download-client/route.ts`**

Create a GET handler:

1. Read query param: `widgetId` (required)
2. Load widget config from DB
3. Extract: `serviceUrl`, `secretName`, `serviceType` ("qbittorrent" or "sabnzbd")
4. For SABnzbd: also extract `showCompleted` (boolean) from config
5. For qBittorrent: also extract `username` from config; the secret holds the password

**qBittorrent handler**:
- The auth flow for qBittorrent is different from other services. Do NOT use fetchService for the login step. Instead:
  1. Resolve the password from secrets: `resolveSecret(secretName)`
  2. Read `username` from config (default "admin")
  3. POST to `${serviceUrl}/api/v2/auth/login` with body `username=${username}&password=${password}` and Content-Type `application/x-www-form-urlencoded`
  4. Extract the SID cookie from the response Set-Cookie header
  5. If login fails (response body !== "Ok."), return error
  6. GET `${serviceUrl}/api/v2/torrents/info?filter=all` with Cookie header `SID=${sid}`
  7. Map each torrent to DownloadItem:
     - id: `torrent.hash`
     - name: `torrent.name`
     - status: map torrent.state to normalized status:
       - "downloading"/"forcedDL" -> "downloading"
       - "pausedDL"/"pausedUP" -> "paused"
       - "queuedDL"/"queuedUP" -> "queued"
       - "uploading"/"forcedUP" -> "seeding"
       - "stalledDL"/"stalledUP" -> "stalled"
       - "checkingDL"/"checkingUP"/"checkingResumeData" -> "checking"
       - "error"/"missingFiles" -> "failed"
       - default -> "queued"
     - progressPercent: `torrent.progress * 100`
     - sizeTotal: `torrent.size`
     - sizeRemaining: `torrent.size * (1 - torrent.progress)`
     - downloadSpeed: `torrent.dlspeed`
     - uploadSpeed: `torrent.upspeed`
     - eta: format `torrent.eta` seconds to "Xh Ym" (if eta > 0 and eta < 8640000), else ""
     - addedAt: convert `torrent.added_on` unix timestamp to ISO string
  8. Build summary: sum dlspeed across all torrents for totalSpeed, count items, count those with state containing "DL" or "downloading" for activeItems

**SABnzbd handler**:
- Use fetchService with authType "query-apikey":
  ```
  fetchService({ baseUrl: serviceUrl, apiKey, endpoint: "/api", authType: "query-apikey",
    queryParams: { mode: "queue", output: "json" } })
  ```
- Map each slot from `response.queue.slots`:
  - id: `slot.nzo_id` (SABnzbd's internal ID)
  - name: `slot.filename`
  - status: map `slot.status` ("Downloading"/"Paused"/"Queued") to lowercase enum
  - progressPercent: `parseFloat(slot.percentage)` or `100 - (slot.mbleft / slot.mb * 100)`
  - sizeTotal: `slot.mb * 1024 * 1024` (convert MB to bytes)
  - sizeRemaining: `slot.mbleft * 1024 * 1024`
  - downloadSpeed: 0 (overall speed is in summary)
  - eta: `slot.timeleft` (already formatted by SABnzbd)
- Build summary from `response.queue`: speed field, slots.length for totalItems, count Downloading for activeItems
- If `showCompleted` is true, also fetch `/api?mode=history&output=json&limit=10` and append completed items with status "completed" or "failed"

Return DownloadClientResponse. Cache-Control: no-store.

Helper to format seconds to duration:
```typescript
function formatEta(seconds: number): string {
  if (seconds <= 0 || seconds >= 8640000) return ""
  const h = Math.floor(seconds / 3600)
  const m = Math.floor((seconds % 3600) / 60)
  if (h > 0) return `${h}h ${m}m`
  return `${m}m`
}
```
