<objective>
Build the Media Requests widget that shows pending and recent media requests from Overseerr or Jellyseerr.

Purpose: Let homelab operators see what content their users have requested, along with request status, without opening Overseerr separately.
Output: A server-side API proxy route and a client widget component for media requests.
</objective>

Infrastructure from Plan 01:
- `fetchService(options)` handles authenticated HTTP requests
- `getServiceConnection(config)` resolves serviceUrl + apiKey
- Overseerr/Jellyseerr use "header-x-api-key" auth type

Overseerr/Jellyseerr API reference (identical APIs since Jellyseerr is an Overseerr fork):
- Request list: GET /api/v1/request?take=20&skip=0&sort=added&filter={status}
  - filter values: "all", "pending", "approved", "available", "processing", "unavailable"
  - Response: { pageInfo: { pages, pageSize, results, page }, results: [...] }
  - Each request object:
    - id, status (1=pending, 2=approved, 3=declined, 4=available, 5=processing)
    - type: "movie" or "tv"
    - media: { tmdbId, tvdbId, mediaType, status }
    - requestedBy: { displayName, avatar }
    - createdAt: ISO date string
    - updatedAt: ISO date string
    - For movies: media.title, media.releaseDate
    - For TV: media.name, media.firstAirDate
  - NOTE: The request object has limited media info. To get title, a separate call to /api/v1/movie/{tmdbId} or /api/v1/tv/{tmdbId} is needed. However, the request results DO include a `media` object with a `status` field and the nested media info varies by version. The safest approach: fetch /api/v1/request which includes mediaInfo, and the title may be in request.media.title or we need to handle it.
  - SIMPLER APPROACH: Use the request endpoint with proper includes. Overseerr v1.33+ returns media details inline. If title is missing, use "Unknown Title (TMDB: {id})".

- Request count by status: GET /api/v1/request/count
  - Response: { pending, approved, processing, available, total }

Existing widget patterns:
- "use client" components with { widgetId, config } props
- Visual: rounded-lg border border-border bg-card, header with icon + title

**Types file: `app/api/widgets/media-requests/types.ts`**

```typescript
export interface MediaRequest {
  id: number
  title: string
  mediaType: "movie" | "tv"
  status: "pending" | "approved" | "declined" | "available" | "processing"
  requestedBy: string
  requestedAt: string     // ISO date string
  year?: number
}

export interface RequestCounts {
  pending: number
  approved: number
  processing: number
  available: number
  total: number
}

export interface MediaRequestsResponse {
  requests: MediaRequest[]
  counts: RequestCounts
}
```

**API route: `app/api/widgets/media-requests/route.ts`**

Create a GET handler:

1. Read query param: `widgetId` (required), `filter` (optional, default "all")
2. Load widget config from DB
3. Extract: `serviceUrl`, `secretName`, `serviceType` ("overseerr" or "jellyseerr" -- they use the same API)
4. Use `getServiceConnection(config)` to resolve API key
5. If not configured, return 400

**Fetch requests**:
- Endpoint: `/api/v1/request?take=20&skip=0&sort=added&filter=${filter}`
- Auth type: `"header-x-api-key"`
- Map status integer to string:
  - 1 -> "pending"
  - 2 -> "approved"
  - 3 -> "declined"
  - 4 -> "available"
  - 5 -> "processing"
- Map each request result:
  - id: `request.id`
  - title: Try `request.media?.title ?? request.media?.name ?? "Unknown Title"`
    - Overseerr/Jellyseerr request results have nested media info but the exact shape varies. Check for `request.media?.title`, then `request.media?.name`, then `request.media?.originalTitle`. As a fallback use `"Request #${request.id}"`.
  - mediaType: `request.type === "movie" ? "movie" : "tv"`
  - status: mapped from integer
  - requestedBy: `request.requestedBy?.displayName ?? request.requestedBy?.email ?? "Unknown"`
  - requestedAt: `request.createdAt`
  - year: try to extract from `request.media?.releaseDate` or `request.media?.firstAirDate` (parse year from date string)

**Fetch counts** (in parallel with requests):
- Endpoint: `/api/v1/request/count`
- Auth type: `"header-x-api-key"`
- Response maps directly to RequestCounts

Return MediaRequestsResponse. If counts fetch fails, return zeroed counts. Cache-Control: no-store.
