Build the Home Assistant widget displaying entity states and providing basic toggle controls for switches and lights.

Purpose: Let homelab operators see smart home entity states at a glance and control basic on/off devices directly from the dashboard.
Output: Home Assistant types, API proxy route (GET + POST), and widget component.
</objective>

Existing service client at lib/services/service-client.ts:
- fetchService<T>(options) with ServiceAuthType

Phase 8 widget pattern (follow exactly):
- Types: app/api/widgets/{name}/types.ts
- Route: app/api/widgets/{name}/route.ts with GET handler

Home Assistant API:
- All states: GET /api/states
  Returns: array of { entity_id, state, attributes: { friendly_name, ... }, last_changed, last_updated }
- Single entity: GET /api/states/{entity_id}
  Returns: { entity_id, state, attributes: { friendly_name, ... }, last_changed }
- Call service: POST /api/services/{domain}/{service}
  Body: { entity_id: "switch.living_room" }
  Common: domain="switch" service="toggle", domain="light" service="toggle"
- Auth: Authorization: Bearer {long-lived access token}

Entity ID format: "{domain}.{object_id}" e.g., "light.living_room", "switch.garage", "sensor.temperature"
Domains that support toggle: light, switch, fan, cover, lock, media_player

  <action>
**Create `app/api/widgets/home-assistant/types.ts`:**

Define normalized types:
- `EntityState`: entityId (string), friendlyName (string), state (string), domain (string), lastChanged (string, ISO date), attributes (Record<string, unknown>), controllable (boolean)
- `HomeAssistantResponse`: entities (EntityState[])
- `ServiceCallRequest`: entityId (string), domain (string), service (string)

**Create `app/api/widgets/home-assistant/route.ts`:**

**GET handler** following Phase 8 pattern:
1. Extract widgetId, validate, call getWidgetWithConfig
2. Call getServiceConnection for serviceUrl and apiKey
3. Read entityIds from config (string array of entity IDs to track)
4. If entityIds is empty or not set, fetch ALL states from GET /api/states using fetchService with authType "header-bearer", then return the first 20 entities (so user can see what's available)
5. If entityIds is provided, fetch GET /api/states with authType "header-bearer", then filter to only the configured entity IDs
6. Map raw HA state objects to EntityState:
   - entityId: raw.entity_id
   - friendlyName: raw.attributes?.friendly_name ?? raw.entity_id
   - state: raw.state
   - domain: raw.entity_id.split(".")[0]
   - lastChanged: raw.last_changed
   - attributes: raw.attributes
   - controllable: domain is one of ["light", "switch", "fan", "cover", "lock", "media_player"]
7. Return HomeAssistantResponse with Cache-Control: no-store

**POST handler** for service calls:
1. Extract widgetId from query params, validate, call getWidgetWithConfig
2. Read request body as ServiceCallRequest
3. Call getServiceConnection for serviceUrl and apiKey
4. Proxy to Home Assistant: fetchService with method "POST", endpoint `/api/services/${domain}/${service}`, authType "header-bearer", body: { entity_id: entityId }
5. Return success/error response

Define raw HA API interface inline:
- HaStateRaw: { entity_id, state, attributes: { friendly_name?: string, [key: string]: unknown }, last_changed, last_updated }
