<objective>
Build the Proxmox VE widget showing node status, VM/container inventory, and resource usage across a Proxmox cluster.

Purpose: Give homelab operators a quick overview of their virtualization infrastructure — which nodes are up, which VMs/containers are running, and how resources are being consumed.
Output: Proxmox types, API proxy route, and widget component.
</objective>

<context>
@.planning/PROJECT.md
@.planning/ROADMAP.md
@.planning/STATE.md

Existing service client at lib/services/service-client.ts:
- fetchService<T>(options) with ServiceAuthType
- getServiceConnection(config) returns { serviceUrl, apiKey }
- Plan 01 adds "header-pve-token" auth type: sets Authorization: PVEAPIToken={token}

Phase 8 widget pattern (follow exactly):
- Types: app/api/widgets/{name}/types.ts
- Route: app/api/widgets/{name}/route.ts with GET handler
- Widget: components/widgets/{name}-widget.tsx "use client" with setup/configured states

Proxmox VE API:
- Nodes: GET /api2/json/nodes
  Returns: { data: [{ node, status, cpu, maxcpu, mem, maxmem, uptime, ... }] }
- Node VMs: GET /api2/json/nodes/{node}/qemu
  Returns: { data: [{ vmid, name, status, cpu, cpus, mem, maxmem, uptime, ... }] }
- Node containers: GET /api2/json/nodes/{node}/lxc
  Returns: { data: [{ vmid, name, status, cpu, cpus, mem, maxmem, uptime, ... }] }
- Auth: API Token in Authorization header: "PVEAPIToken=USER@REALM!TOKENID=UUID"
  The user stores the full token string (e.g., "user@pam!mytoken=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx") as a secret

Important: Proxmox API wraps all responses in { data: [...] }. The route must unwrap this.
</context>

  <action>
**Create `app/api/widgets/proxmox/types.ts`:**

Define normalized types:
- `ProxmoxNode`: id (string), name (string), status ("online" | "offline"), cpuUsage (number, 0-100), memUsage (number, 0-100), memTotal (number, bytes), uptime (number, seconds)
- `ProxmoxGuest`: vmid (number), name (string), node (string), type ("qemu" | "lxc"), status ("running" | "stopped" | "paused"), cpuUsage (number, 0-100), memUsage (number, 0-100), memTotal (number, bytes), uptime (number, seconds)
- `ProxmoxResponse`: nodes (ProxmoxNode[]), guests (ProxmoxGuest[]), summary: { totalNodes, onlineNodes, totalGuests, runningGuests }

**Create `app/api/widgets/proxmox/route.ts`:**

GET handler following the Phase 8 pattern:
1. Extract widgetId, validate, call getWidgetWithConfig
2. Call getServiceConnection for serviceUrl and apiKey
3. Fetch nodes: fetchService with endpoint "/api2/json/nodes", authType "header-pve-token"
   - Proxmox wraps responses in { data: [...] }, so the generic type should be { data: ProxmoxNodeRaw[] }
4. For each online node, fetch VMs and containers in parallel:
   - VMs: fetchService endpoint "/api2/json/nodes/{nodeName}/qemu", authType "header-pve-token"
   - LXC: fetchService endpoint "/api2/json/nodes/{nodeName}/lxc", authType "header-pve-token"
5. Map raw Proxmox node data to ProxmoxNode:
   - status: node.status === "online" ? "online" : "offline"
   - cpuUsage: node.cpu * 100 (Proxmox returns 0-1 float)
   - memUsage: (node.mem / node.maxmem) * 100
   - memTotal: node.maxmem
   - uptime: node.uptime
6. Map raw VM/LXC data to ProxmoxGuest:
   - type: "qemu" for VMs, "lxc" for containers
   - cpuUsage: guest.cpu * 100 (0-1 float from Proxmox)
   - memUsage: (guest.mem / guest.maxmem) * 100
   - For stopped guests, cpuUsage and memUsage are 0
7. Compute summary counts
8. Return ProxmoxResponse with Cache-Control: no-store

Define raw Proxmox API interfaces inline in the route file (not exported):
- ProxmoxNodeRaw: { node, status, cpu, maxcpu, mem, maxmem, uptime, ... }
- ProxmoxGuestRaw: { vmid, name, status, cpu, cpus, mem, maxmem, uptime, ... }
  </action>
