<objective>
Build the Uptime Kuma widget showing monitor status and uptime percentages using the public status page API.

Purpose: Display service monitoring status at a glance — which monitors are up, which are down, and overall uptime percentages. Uses publicly accessible APIs to avoid the complexity of Uptime Kuma's WebSocket-based authenticated API.
Output: Uptime Kuma types, API proxy route, and widget component.
</objective>

Existing service client at lib/services/service-client.ts:
- fetchService<T>(options) — can be used without auth by passing empty apiKey and any authType (the auth header just adds an empty value, which is harmless for public endpoints)

Phase 8 widget pattern (follow exactly):
- Types: app/api/widgets/{name}/types.ts
- Route: app/api/widgets/{name}/route.ts with GET handler
- Widget: components/widgets/{name}-widget.tsx "use client" with setup/configured states

Uptime Kuma API (public endpoints, no auth required):
- Status page: GET /api/status-page/{slug}
  Returns: { config: { name, ... }, publicGroupList: [{ id, name, monitorList: [{ id, name, type, ... }] }] }
- Heartbeat: GET /api/status-page/heartbeat/{slug}
  Returns: { heartbeatList: { "{monitorId}": [{ status, time, ping, msg }] }, uptimeList: { "{monitorId}_24": 0.99, "{monitorId}_720": 0.98 } }
  status values: 0=down, 1=up, 2=pending, 3=maintenance

- Prometheus metrics: GET /metrics (if enabled in Uptime Kuma settings, 1.23+)
  Format: Prometheus text format
  Key metrics:
    monitor_status{monitor_name="X",monitor_type="Y",monitor_url="Z"} 1
    monitor_response_time{...} 123

Important design notes:
- Uptime Kuma's primary API is WebSocket-based (socket.io) and requires authentication — this is too complex for a dashboard widget
- The status page API and Prometheus /metrics endpoint are the only practical REST endpoints
- Status page approach requires the user to have created at least one status page in Uptime Kuma
- The widget should clearly document this requirement in the settings helper text
</context>

**Create `app/api/widgets/uptime-kuma/types.ts`:**

Define normalized types:
- `MonitorStatus`: "up" | "down" | "pending" | "maintenance"
- `Monitor`: id (number), name (string), status (MonitorStatus), uptime24h (number, 0-100 percentage), uptime30d (number, 0-100 percentage), responseTime (number, ms), url (string, optional)
- `UptimeKumaResponse`: monitors (Monitor[]), summary: { total, up, down, pending, maintenance }, statusPageName (string)

**Create `app/api/widgets/uptime-kuma/route.ts`:**

GET handler:
1. Extract widgetId, validate, call getWidgetWithConfig
2. Read config: serviceUrl, slug, dataSource ("status-page" or "metrics", default "status-page")
3. Validate serviceUrl is present. slug is required if dataSource is "status-page"

**Status page approach** (dataSource === "status-page"):
4. Fetch status page config: plain fetch (no auth needed) to `${serviceUrl}/api/status-page/${slug}` with AbortSignal.timeout(10_000)
   - Do NOT use fetchService here since no auth is needed — just use plain fetch with timeout
   - Parse JSON response
5. Fetch heartbeat data: plain fetch to `${serviceUrl}/api/status-page/heartbeat/${slug}`
   - Parse JSON response
6. Combine data:
   - From status page response: extract monitors from publicGroupList[].monitorList[]
   - From heartbeat response: match heartbeatList[monitorId] for latest status and uptimeList for uptime percentages
   - Map to Monitor[]:
     - id: monitor.id
     - name: monitor.name
     - status: latest heartbeat status (0=down, 1=up, 2=pending, 3=maintenance)
     - uptime24h: uptimeList["{id}_24"] * 100
     - uptime30d: uptimeList["{id}_720"] * 100
     - responseTime: latest heartbeat ping value
     - url: monitor.url if present

**Metrics approach** (dataSource === "metrics"):
4. Fetch plain text from `${serviceUrl}/metrics` with AbortSignal.timeout(10_000)
5. Parse Prometheus text format:
   - Extract `monitor_status{monitor_name="X",...} VALUE` lines
   - Extract `monitor_response_time{monitor_name="X",...} VALUE` lines
   - For each unique monitor_name, create a Monitor with:
     - status: value 1 = "up", 0 = "down"
     - responseTime: from monitor_response_time
     - uptime24h/uptime30d: not available from /metrics, set to -1 to indicate unavailable

6. Compute summary counts
7. Return UptimeKumaResponse with Cache-Control: no-store

Define raw API response interfaces inline for the status page and heartbeat responses.
